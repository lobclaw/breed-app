# 功能设计：首页、提醒、状态、协作、导航

## 1. 首页设计

### 1.1 固定结构

首页当前固定为两层：

1. 顶部固定区：`逾期 / 繁育 / 用药 / 健康` pills + `WeekStrip`
2. 正文任务区：`逾期待处理` → `繁育流程` → `今日用药` → `健康提醒`

规则：

- 过去日期灰色不可点击
- 今天与未来日期可点击
- 有任务或疗程状态的日期显示红点
- 首页红点和区块计数按全部事项数计算，不随折叠后的可见卡片数变化
- 点击月份标题打开首页任务日历；任务日历只用于选日期，不展示时刻
- 从任务日历选中未来日期后，`WeekStrip` 切到该日期所在周；顶部副标题仅今天显示“今日概览”，非今天显示“当日安排”
- `WeekStrip` 顶部日期区采用导航条表达：左侧月份主入口，右侧固定日历按钮；非今天态再显示轻量 `今日` 入口，一键回到首页默认今日视图
- 当前不做摘要卡、统一任务池、二级页或通用 Sheet

### 1.2 卡片组织原则

首页不是纯犬视角，也不是纯事件视角，而是按“注意力单元”组织：

- 单犬需要单独推进：单犬卡/单行
- 同类健康事项适合一起做：批量卡
- 疗程状态需要当天盯住：今日用药卡
- 繁育是流程推进器：按步骤组织

### 1.3 当前区块规则

#### 逾期

- 只承接已过期且仍待处理的健康任务与繁育额外安排
- 颜色使用强 red
- 优先级最高

#### 繁育

- 主链固定：`发情 → 建议卵泡检查 → 配种 → 建议孕检 → 生产 → 确认断奶`
- `breeding_milestone` 必须按 `details.step_type` 路由
- 首页 `建议卵泡检查` 摘要固定为“发情第 X 天 + 最近一次检查摘要 + 最近一次异常提示”
- 繁育主链 milestone 在首页按过程状态卡展示，不按普通待办卡展示
- 建议日期只用于流程判断参考，不升级为逾期待办
- 主动作统一为 `处理`，真实关闭依赖记录页提交成功
- 自由录入候选：`发情` 隐藏发情中/怀孕中/哺乳中种母；`卵泡检查` 与 `配种` 显示未配种种母（空闲与发情中），隐藏怀孕中/哺乳中；当前发情中显示 `发情第 X 天`，无当前发情状态但有历史记录时显示 `上次发情：YYYY-MM-DD`
- 繁育周期详情页入口按语义分工：`+ 添加记录` 只放记录/向导，怀孕中包含孕检、产检、临产监测、录入生产、异常终止记录；`... 更多操作` 只放周期级管理动作
- 当前周期已经存在 `pregnancy_check` 后，`+ 添加记录` 不再展示 `孕检记录`；后续复查走 `产检记录`，历史孕检从时间线进入详情或编辑
- `放弃本次繁育` 是独立周期管理动作，所有未终态周期均可在 `... 更多操作` 中执行，直接将周期标记为 `放弃`，不创建繁育记录
- `异常终止记录` 表单按当前周期状态过滤类型：`发情中` 只显示 `放弃配种`，提交后创建记录并将周期标记为 `放弃`；`怀孕中` 显示 `流产 / 死胎 / 医疗终止 / 未怀孕`，提交后创建记录并将周期标记为 `失败`
- 额外安排独立成层，不与主链混合
- 繁育不是批量完成模型

#### 健康

- 只展示已明确创建的健康提醒
- 疫苗/驱虫优先保留批量价值
- 疫苗/驱虫按事项卡呈现，即使只有 1 只犬也使用批量卡结构，不回落到单犬健康卡
- 疾病观察留在健康区，不并入用药

#### 用药

- 以 `medication_tasks` 为事实源
- 区分 `待完成 / 部分完成 / 已完成 / 漏服`
- 即使过了当天也留在用药区，不抬升到逾期区

## 2. 首页视觉与动作口径

### 2.1 颜色语义

- `逾期 = 强 red`
- `繁育 = 繁育色`
- `健康 = blue`
- `用药 = plum`

健康二级类型：

- `疫苗 = blue`
- `驱虫 = teal`
- `疾病 = 柔和 red`
- `用药执行 = plum`

### 2.2 卡头结构

- 标题只表达卡片类型
- 副标题表达对象规模，如 `12只犬`
- 右侧角标表达项数或进度

固定口径：

- `疾病观察`：标题=`疾病观察`，副标题=`X只犬`，角标=`X项`
- 疫苗/驱虫批量卡：`groupTitle` 只保留业务标题，数量写副标题

### 2.3 动作命名与层级

- 单条入口统一叫 `处理`
- 集合入口统一叫 `批量操作`
- 主 CTA 是实底按钮
- 次 CTA 是浅底或描边按钮
- 弱动作只用文本或弱 chip

行内 `处理 / 康复 / 停药` 统一视为次级动作，不能压过底部主 CTA。
繁育主链不提供 `推迟 / 跳过`，仅额外安排保留任务态动作。

## 3. 提醒系统

### 3.1 任务来源

| 来源 | 首页归属 | 规则 |
|------|----------|------|
| 繁育主链里程碑 | 繁育 | 自动推进 |
| 繁育额外安排 | 繁育 | 手动附加，不改变主流程 |
| 疫苗/驱虫待办 | 健康 | 仅显式创建才出现 |
| 疾病观察 | 健康 | 保持健康语义 |
| 连续用药疗程 | 用药 | 由 `medication_tasks` 提供 |
| 已过期待办 | 逾期 | 从繁育/健康升级而来 |

### 3.2 `tasks` 集合职责

`tasks` 负责：

- 待办与提醒展示
- 完成、推迟、跳过
- 首页计数与日期红点的基础数据

它不负责：

- 连续用药真实状态
- 犬只当前状态
- 真实健康/繁育记录本体

补充规则：

- `breeding_milestone` 在首页按过程状态解释，而不是普通待办
- `breeding_extra_arrangement` 继续保留任务语义，可完成、推迟、跳过

### 3.3 连续用药

- 连续用药使用 `medication_tasks`
- 用药优先从疾病记录发起，并写入 `source_record_id` 关联对应疾病；仍允许独立创建
- 首页与详情页共用同一套“漏服”判断
- `完成` 是主 CTA，`批量操作` 是次 CTA
- `批量康复` 等集合动作收纳进 `今日用药批量操作` Sheet
- 犬只详情健康 Tab 的 `用药` 分组展示全部用药疗程，`进行中` 置顶，其次 `已完成`，最后 `已取消`

## 4. 状态系统

### 4.1 自动推导状态

从记录实时推导：

- 发情
- 怀孕
- 哺乳
- 疾病
- 用药

### 4.2 详情与列表显示口径

- 犬只详情“当前状态”不是时间线，而是当前生效状态集合
- 显示优先级：`疾病` → `用药` → `怀孕/哺乳` → `发情`
- Hero 区与正文列表复用同一排序结果
- Hero 区继续使用 `用药中` 当前状态标签；健康 Tab 内的 `用药` 分组承担全部历史疗程浏览
- 疾病显示需去重；疾病主文案使用主疾病，症状以弱文案补充
- 列表疾病摘要最多一个 tag，多病名做聚合摘要
- `外部种公` 详情页按轻量档案展示，只保留 `概览 / 繁育`；不展示健康、用药、体重、财务、普通添加记录、当前健康状态、可用状态和去向管理入口
- `外部种公` 的繁育页用单一“使用记录”列表追踪作为种公参与的配种周期，并在同一张卡中合并展示产窝结果，核心目标是联系人/来源可查、使用记录可追溯

## 5. 协作权限

### 5.1 角色

V1 保持三级角色：

- `creator`
- `admin`
- `helper`

### 5.2 原则

- 单家庭模式
- 所有写操作通过云对象 `_before` 注入 `familyId`
- 列表与详情展示需要保留操作归因

### 5.3 通知设置

- 通知设置统一挂在 `families.settings`
- 包含：`push_enabled`、`morning_summary_enabled`、`morning_summary_time`、`notification_types`
- `notification_types` 当前固定为：`breeding / vaccination / medication / care_group / overdue`
- `overdue` 不可关闭，后端兜底保持为开启
- `care_group` 仅保留配置与数据结构，未接入自动任务链路，入口默认隐藏，不按已上线能力验收
- 晨间摘要按家庭设置的北京时间 `HH:MM` 分钟级命中
- 总开关或晨间摘要开关关闭时，不发送晨间摘要
- 晨间摘要内容按通知类型过滤：繁育、疫苗/驱虫、今日用药、护理提醒、逾期

### 5.4 数据备份与导出

- `立即备份` 生成云端 JSON 备份文件，并更新 `families.settings.last_backup_date / last_backup_file_id`；成功后只提示云端备份完成，不触发本地下载
- “立即备份/自动备份”只保留最近 4 份备份文件，新备份成功后服务端删除更旧备份；普通导出不计入备份保留份数
- 备份历史展示最近 4 份备份；可下载备份文件，且仅支持从最近备份列表恢复
- 恢复备份由云对象执行：恢复前先自动创建一份恢复前备份，再用备份内业务集合覆盖当前家庭业务数据；账号、家庭成员、操作日志与同步 mutation 不恢复
- `导出到本地` 生成用户可下载/分享的数据归档并提供下载链接；JSON 为完整归档，CSV 为按集合拆分的 zip
- 备份与导出执行前必须确认本地无待同步、同步失败、冲突或待上传数据
- `auto_backup_enabled` 表示每周自动备份开关；实际触发由云端定时任务执行
- `数据修复` 只自动修复低风险技术字段；业务风险问题只返回告警，不自动改业务状态

## 6. 导航结构

### 6.1 底部导航

当前主要业务域：

- 首页
- 犬只
- 繁育
- 财务
- 设置/更多

### 6.2 关键入口约束

- 收入统一从 `pages/finance/expense-add.vue?type=income` 进入
- 旧 `income-add.vue` 已移除
- 从首页进入繁育流程时必须带 `dogId + dogName + cycleId + taskId + locked=true`

### 6.3 回收站

- 回收站是“软删除对象恢复中心”，不是业务历史列表
- 当前纳入：`dogs`、`expenses`、`incomes`、`agents`、`medication_protocols`
- 当前不纳入：`sale_records`，因为其删除/恢复还会联动自动收入与犬只去向
- 回收站按混合列表展示，按删除时间倒序排序
- 每项可执行 `恢复` 或 `永久删除`
- 统一保留 30 天，倒计时按 `deleted_at` 与当前时间实时计算

## 7. 关键业务细则

### 7.1 健康提醒

- 疫苗/驱虫记录默认只计算建议日期
- 疫苗候选列表不展示繁育阶段；有历史疫苗记录的犬只在名字后显示 `上次疫苗：YYYY-MM-DD`
- 驱虫候选列表不展示繁育阶段；有历史驱虫记录的犬只在名字后显示 `上次驱虫：YYYY-MM-DD`
- 疾病记录与用药任务的犬只候选列表在名字右侧展示当前疾病与当前用药标签，标签文案和颜色沿用档案列表状态口径
- 只有显式勾选「创建下次待办」才生成任务
- 同一犬只在同一北京时间自然日内，重复录入同子类型疫苗/驱虫时应跳过并提示
- 若同一犬只当天已存在同子类型疫苗/驱虫真实记录，则不应再创建同日同子类型待办
- 首页批量完成疫苗/驱虫时必须同时创建真实 `health_record`

### 7.2 额外安排

- 繁育表单可附加 `额外安排`
- 字段仅 `kind + due_date + notes`
- 默认挂在 `cycle_id`
- 完成/推迟/跳过仅影响任务本身

### 7.2a 配种方式

- 配种方式选项顺序统一为 `人工授精` 在前、`自然交配` 在后
- 新建与编辑配种记录默认选中 `人工授精`
- 配种表单展示“第几脚”，不允许手动修改
- `mating_number` 由系统按同周期已有配种记录数自动计算，依次显示 `第1脚 / 第2脚 / 第3脚 ...`

### 7.3 繁育主链首页表达

- 首页主链卡展示当前阶段、距关键起点已过天数、建议日期
- 超过建议日期后只增强文案与颜色，不升级到逾期区
- 进入记录页处理后，由记录提交流程关闭对应 milestone
- 从首页进入繁育主链记录页提交成功后，返回首页应优先保留原卡位并直接更新为下一阶段，不做“先整卡消失再刷新回来”的承接
- 首页繁育组卡主动作保持 `处理`；`建议卵泡检查` 行允许补充弱文本动作 `观察 / 直接配种`，其他上下文快捷入口收纳进 `处理` 弹窗
- 首页繁育组卡与单犬繁育卡统一：点击头像进入犬只详情；点击卡片主体进入繁育详情；点击 `处理` 才进入当前阶段记录页
- `follicle_check` 只有在 `details.result === 已成熟` 时才推进到 `配种`；最近一次结果为 `发育中 / 发育不良 / 其他` 时，首页继续停留在 `建议卵泡检查`
- `breeding_milestone(details.step_type=follicle_check)` 需补齐 `follicle_check_count / follicle_result / latest_follicle_check_date / abnormal_result`
- 未成熟后的 `建议卵泡检查` 默认按“次日复查”挂下一步；其中仅 `发育不良` 增加异常强化提示，不改变首页分区
- `观察` 与 `直接配种` 都不计入任务完成，不改变卡片状态、红点和首页计数；只有记录提交成功后才关闭对应 milestone
- 孕检确认怀孕后，首页 `birth` milestone 仍代表下一硬节点，但阶段标题显示为 `待产`，日期口径显示为 `预产期`
- 首页 `待产` 行外层只保留主动作 `处理`；`处理` 弹窗内提供 `录入生产记录 / 产检 / 临产` 等上下文动作，其中 `产检` 常驻显示，`临产` 仅在距预产期 5 天内显示；`产检 / 临产` 都只作为上下文快捷入口，不关闭主链 milestone
- 生产记录向导 Step 2 的幼崽昵称字段保持 `选填`；若留空，提交时自动写入默认名 `<母犬名>窝-<录入序号>号`
- 生产记录向导 Step 3 不是纯确认页，需包含 `经验心得（选填）` 与 `首次驱虫提醒 / 首次疫苗提醒` 两个显式勾选项
- 生产记录页勾选的 `首次驱虫提醒 / 首次疫苗提醒` 只创建普通健康待办，不生成真实 `health_record`，且只为存活幼崽创建

### 7.4 重复用药检测

- 同狗同药名只允许一个进行中任务
- 同狗同药名只阻止“当前仍有效的进行中疗程”
- 已超过疗程天数的旧任务，在重复检测前自动收口为 `已完成`
- 重复时前端弹窗分“将创建”与“已有同名任务”两段
- 覆盖的语义是取消旧任务后创建新任务
- 任务状态表达疗程是否结束；具体执行完整度通过详情页或副文案说明，不新增主状态

### 7.5 疾病记录唯一性

- 同狗同主疾病且未康复，不允许重复创建
- 命中重复时应优先引导编辑原记录
- 服务端必须兜底校验

### 7.6 财务入口与统计

- 统一记账入口为 `pages/finance/expense-add.vue?type=income`
- 旧 `income-add.vue` 已移除，不再维护第二套收入写路径
- 手动收入类型固定为 `销售 / 定金保留 / 领养 / 其他`
- 财务首页筛选结构固定为“顶部 `全部 / 收入 / 支出` + 统一高级筛选入口”
- 已生效筛选条件必须显式显示为 chips，并支持单个清除与一键清空
- 同维度多选取并集，不同维度取交集
- `仅看无关联` 与任意关联对象互斥
- 收入关联只支持 `犬只 / 无关联`，支出关联支持 `犬只 / 窝 / 繁育周期 / 无关联`
- 自定义支出分类必须带 `parent_group`
- 财务统计页底部模块固定命名为 `专项报表`，不跟随上方时间维度切换
- `未来预估` 禁止先展示硬编码默认值再异步覆盖；合法 `0` 值必须能覆盖默认值
- 窝详情后续添加幼崽按补录生产事实处理，单窝统计与利润读取按当前非删除关联幼崽数纠偏出生/存活展示
- `单窝利润` 中若同一笔 expense 同时关联 `linked_cycle_id` 与 `linked_litter_id`，只统计一次，并优先归入“窝级别费用”

### 7.7 销售流程

- 顶层销售状态保持 `待售 / 已预定 / 已成交 / 已退款 / 定金取消`
- `已成交` 与 `settlement_status` 分离，后者当前使用 `未结算 / 部分结算 / 已结算`
- `待售` 表示已手动纳入销售池，不要求先填写 `floor_price`；`floor_price` 仅表示内部参考底价，可为空
- “开始销售”提交走 `finance-service.createSaleRecord`；“直接成交”是销售创建页的一步式入口，底层仍先创建 `待售` 销售记录，再立即走 `finance-service.completeSale`
- 仅 `幼崽` 且 `disposition` 属于 `在养 / 自留 / 待售` 可开始销售；同一犬只同一时刻只允许一条进行中的销售记录
- 销售创建页的犬只候选必须来自本地投影过滤：`role=幼崽`、`disposition in 在养/自留/待售`，并排除已有 `待售 / 已预定` 销售记录的犬只；锁定来源入口仍显示来源犬只，但提交必须由本地事务和云对象继续兜底校验
- 开始销售必须同时创建 `sale_records.status=待售`，并将犬只 `disposition` 切到 `待售`
- `定金取消` 后旧销售记录作为终态历史保留，犬只回到 `在养`；同犬多条取消历史只在最新一条可重启记录上展示“再次销售”，底层创建新的 `待售` 销售记录，不复用或覆盖旧取消记录
- 销售创建页 `/pages/sale/create` 支持“开始销售 / 直接成交”模式切换，并支持来源页透传 `dogId + dogName` 锁定犬只
- “开始销售”字段固定为：犬只、`sale_mode`（默认待定，落库为 `null`）、`floor_price`（选填）、`buyer_info`（选填）、`notes`（选填）
- “直接成交”字段固定为：犬只、`sale_mode`（不展示待定，默认 `自售`）、`received_amount`（选填）、`agreed_price`（选填）、`buyer_info`（选填）、`platform`（选填）、`seller_agent_id/name`（仅 `代理 / 代卖` 时选填）、`delivery_date`（选填）、`notes`（选填）
- `sale_mode=null` 表示销售方式待定；待售、已预定、已成交阶段允许在详情页单独修改，收定金与完成交易时也可顺手确认或继续保留待定
- 销售列表与详情按新语义展示：`待售` 无底价时显示“未定价”；`已成交` 且无 `received_amount` 时显示“未结算”，不要渲染 `¥0`
- 完成交易走 `finance-service.completeSale`，允许 `received_amount` 为空；为空时写入 `status=已成交`、`settlement_status=未结算`，且不自动生成销售收入
- 成交后补结算统一走 `finance-service.settleSale`；补录 `received_amount` 后再创建或更新自动收入
- 销售详情页需提供“补录结算”入口；未结算成交不得展示或提交退款
- 退款只允许已结算成交，且 `refund_amount` 必须 `> 0` 且不超过 `received_amount`；全额退款后犬只回到 `在养`
- 定金取消以 `deposit_kept_amount` 表达保留金额，必须在 `0..deposit_amount` 内；用户输入退还定金时需先换算为保留金额再提交
- `finance-service.getSaleList/getSaleDetail` 返回值必须走销售归一化口径，稳定包含 `sale_mode`、`settlement_status`、`agent_name`
- 本地 `listLocalSales/getLocalSaleDetail` 也必须走同一销售归一化口径，列表和详情不得出现代理人、结算状态、犬只性别/月龄不一致
- `sale_records` 当前不纳入回收站；未定义自动收入与犬只去向回滚前不要顺手接入

## 8. 首页一致性规则

### 8.1 本地承接

- 提交成功后先本地承接首页变化，再后台刷新
- 批量卡只移除真实完成子集
- 首页批量健康卡逐只点击时，先保留卡片并把对应犬只标记为已完成，同时更新分数与进度条
- 首页批量健康卡只有在全部犬只都完成后，才进入短暂完成态并整卡退场
- 首页批量健康卡底部主按钮保持批量补齐剩余未完成犬只的语义
- 局部承接要同步批量卡元数据

### 8.2 防闪回

- 最近完成的任务加入 suppression
- 服务端刷新先过滤 suppression
- 加载链路必须保留 latest token 保护

### 8.3 WeekStrip 红点

- 今日红点以全部事项数为准，不随首页折叠后的可见卡片数变化
- 不依赖 `counts.today`
- 未来只有用药疗程状态时也必须有红点
- 首页任务日历沿用同一份日期计数缓存；月历中今天与未来日期有任务/疗程状态时显示红点
