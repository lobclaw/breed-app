# 功能设计：首页 + 提醒 + 状态 + 协作 + 导航

## 一、首页设计

### 1.1 三层结构

```
┌─ Zone 1：状态摘要栏（固定，不滚动）────────────────┐
│  [🟡 今日 X]    [本周 X]    [30天 X]                │
│  点击数字 → 滚动到卡片区                             │
├─ WeekStrip（固定，周一到周日）─────────────────────┤
│  一    二    三    [今]   五    六    日              │
│  灰    灰    灰    •      •          •              │
│  过去日期灰色不可点击，只能点击今天和未来日期          │
├─ Zone 2：犬只卡片区（可滚动）───────────────────────┤
│  单列表：逾期任务（红色）在最上面 → 今日任务           │
│  不分区，逾期按逾期天数降序排列                       │
└─────────────────────────────────────────────────────┘
```

Zone 1 数字说明（累进包含关系：今日 ⊂ 本周 ⊂ 30天）：
- 「今日」= 逾期事项 + 今日待办（合并计数）。有逾期时 pill 变红色
- 「本周」= 从今天到本周日的所有待办（含逾期+今日）
- 「30天」= 从今天起未来30天的所有待办（含本周）

WeekStrip（周历条）：
- **周一到周日固定排列**，保持周感（参考 Things 3 / Apple Reminders 面向未来的设计）
- 过去日期灰色不可点击，逾期任务已"飘"到今天列表顶部
- 今天和未来日期可点击，有任务显示红点（Apple Calendar 风格）
- 点击未来日期 → 下方切换为该天任务（从 weekCache 读取，零延迟）
- 点击月份标题展开完整月历

逾期任务混入设计（参考 Apple Reminders / Todoist）：
- 逾期任务不再有独立"需处理"区域，直接混入今日列表最顶部
- 逾期卡片红色边框 + "逾期X天" badge
- 逾期天数越多排越靠前（最紧急的最先看到）
- 逾期任务只出现在今天列表，不在 WeekStrip 过去日期重复显示

### 1.2 智能卡片系统

**卡片组织原则：以「注意力单元」为主，系统自动匹配最合适的展示方式。**

不是纯粹以犬为主，也不是以事件为主，而是匹配繁育者实际的思维方式：
- 照顾某只犬时 → 以犬为中心
- 给全场犬驱虫时 → 以事件为中心
- 给所有怀孕犬喂钙铁时 → 以状态群组为中心

**四种卡片类型：**

**① 个体犬只卡片**

某只犬有独立任务（配种/孕检/卵泡检查等）

```
┌──────────────────────────────────┐
│  大米 · 发情第14天                 │
│  今日打卵泡（已预约）              │
│  [记录结果]                       │
└──────────────────────────────────┘
```

**② 状态群组卡片**

某个状态群组有共同任务

```
┌──────────────────────────────────┐
│  🤰 孕期护理 · 杠上花、安妮、小乔  │
│  每日喂钙铁                       │
│  杠上花：怀孕第3天                 │
│  安妮：怀孕第18天                  │
│  小乔：怀孕第45天                  │
│  [批量标记今日已完成]              │
└──────────────────────────────────┘
```

**③ 批量卡片**

同类事件 + 同天 + 2只以上犬

```
┌──────────────────────────────────┐
│  💉 月度驱虫 · 今日 · 12只成犬     │
│  小白、花花、豆豆、妞妞 等12只     │
│  [批量处理全部]  [逐只查看]        │
└──────────────────────────────────┘
```

全部犬只时：「月度驱虫 · 今日 · 全部成犬(35只)」

**④ 健康关注卡（升级自连续用药卡片）**

统一展示所有需要健康关注的犬只：生病中 + 用药中 + 生病且用药中。
始终显示在今日列表中（不标记为逾期）。

三种行状态：
```
┌──────────────────────────────────────────────┐
│  🩺 健康关注 · 3只犬                    0/2    │
│  🔴 肉肉  感冒 · 阿莫西林 50mg 3/5天 口服×2  │ ← sick_with_med（有 checkbox）
│  🔴 1号   感冒 · 观察中 · 第3天              │ ← sick_only（无 checkbox）
│  ☐  大桥  益生菌 2ml 1/3天 口服              │ ← med_only（有 checkbox）
│  [完成]  [推迟]                              │
└──────────────────────────────────────────────┘
```

- sick_only：犬只生病但未用药（观察期），红色圆点 + 病症 + 治疗状态 + 第X天
- sick_with_med：犬只生病且在用药，checkbox + 病症 + 药品全信息
- med_only：犬只仅用药未生病，checkbox + 药品进度
- 数据来源：health_records（type=illness, treatment_status≠已康复）+ medication daily tasks
- 逾期的用药 task 由每日审计标记为 skipped，不影响首页展示

**卡片合并规则：**

| 条件 | 展示方式 |
|------|---------|
| 同类事件 + 同天 + 1只犬 | 独立的犬只卡片 |
| 同类事件 + 同天 + 2只以上 | 合并为一张批量卡片 |
| 同窝幼崽的同类事件 | 合并为一张窝卡片 |
| 同一只犬的多个任务 | 合并为一张犬只卡片，按优先级排任务 |
| 同一只犬既有个体任务又属于批量/群组任务 | 允许出现两次（方案A） |

**合并优先级（从高到低）：**

1. **窝级别合并**（最高优先）：同窝幼崽的同类事件 → 合并为窝卡片
2. **批量合并**：非同窝、同类事件 + 同天 + 2只以上 → 合并为批量卡片
3. **个体卡片**：不满足以上条件的 → 各自独立卡片

**冲突场景示例：**
- 花花第3窝5只幼崽今天都需要打疫苗 → 按规则1，合并为「窝卡片」（不走规则2的批量卡片）
- 3只不同窝的成犬今天都需要驱虫 → 按规则2，合并为「批量卡片」
- 花花第3窝3只 + 豆豆第2窝2只今天都需要疫苗 → 各窝分别合并为窝卡片（两张卡片），不跨窝合并

**单卡任务数限制：**
- 单张犬只卡片最多显示 3 个任务，超出显示「还有X项」
- 批量卡片最多显示 12 只犬，超出分页或显示「还有X只」

**一只犬只有多任务时，只出现一张卡片，放在最高优先级区域：**

```
（逾期+今日混入同一列表，逾期在最上面）
┌──────────────────────────────────┐
│  🔴 逾期2天                       │
│  小白 · 马尔济斯 · 疫苗            │
│  [完成]  [推迟]                    │
└──────────────────────────────────┘
┌──────────────────────────────────┐
│  小白 · 马尔济斯 · 今日体检        │
│  [完成]  [推迟]                    │
└──────────────────────────────────┘
```

### 1.3 卡片核心设计规则

**以犬为主语，不是以任务为主语：**
- 「花花 · 孕期第58天」而非「孕检任务」
- 同一只犬的多个任务合并为一张卡片
- 同一窝的批量任务合并（「豆豆第3窝 5只幼崽 · 首次驱虫」）

**状态语言优于日期语言：**
- 「孕期第58天 · 距预产期5天」而非「3月26日预产」
- 「发情第12天 · 配种窗口期」而非「3月21日建议配种」

**卡片操作按钮设计（渐进式披露）：**

任务分两大类，完成策略不同：
- **健康类**（疫苗/驱虫/用药）：task 里已有完整信息（犬只/类型/药品），可一键自动创建记录
- **繁育类**（发情/繁育里程碑/疾病复查）：需要填写观察细节，必须进入表单

操作矩阵：

| 卡片类型 | 点击卡片 | checkbox | 完成 | 推迟 | 跳过 |
|---------|---------|---------|------|------|------|
| 单犬·健康 | 跳转表单（高级） | 无 | ✅ 自动创建记录+完成+生成下次提醒 | ✅ | ✅ 仅标记done |
| 单犬·繁育 | 跳转表单 | 无 | 无（必须进表单） | ✅ | ✅ 仅标记done |
| 批量·健康 | 跳转批量表单 | ✅ 勾=自动完成该犬 | ✅ 全部未勾选的一键完成 | ✅ | ✅ 全部标记done |

按钮优先级（从左到右）：完成（主按钮）> 推迟 > 跳过（文字按钮）
所有操作完成后有明确视觉反馈（卡片消失动画 + pill 数字更新）
触控目标 ≥ 44pt（适配脏手/湿手场景）

**"完成"按钮的后端行为（健康类）：**
1. 读取 task 的 type + details（vaccine_type/deworming_type 等）
2. 自动创建 health_record（用 task 数据 + 今天日期）
3. 调用 generateReminders 生成下次提醒
4. 标记 task 为 completed

**逾期是提醒，不是审判：**
- 逾期卡片红色边框 + "逾期X天" badge，混入今日列表最顶部
- 推迟时必须选择新的提醒日期（默认明天）
- 可选填原因（如「等兽医回复」）
- 推迟后移出列表，到新日期再出现

**批量任务显示进度：**

```
┌──────────────────────────────────┐
│  豆豆第3窝 · 首次驱虫 · 3/5只待处理│
│  ✅ 幼崽A ✅ 幼崽B                │
│  ⬜ 幼崽C  ⬜ 幼崽D  ⬜ 幼崽E     │
│  [批量处理剩余3只]                │
└──────────────────────────────────┘
```

**协作透明：**
- 已完成任务显示操作人和时间：「小李已完成 · 14:30」

**卡片数量控制：**
- 每个优先级区域最多显示 6-8 张卡片
- 超出部分折叠为「还有 X 件事项 ▾」
- 0 件事时：显示「犬群一切正常 · 距下一个事项还有 X 天」

## 二、提醒/通知系统

### 2.1 提醒来源（全部自动生成，不需手动创建）

| 来源 | 提醒内容 | 推算逻辑 |
|------|---------|---------|
| 发情记录 | 「小白：发情第X天，建议查卵泡」 | 发情开始日期 + 可配置天数 |
| 配种记录 | 「小白：预计孕检日」 | 配种日期 + 25~30天（可配置） |
| 配种记录 | 「小白：预产期还有X天」 | 配种日期 + 63天 |
| 临产体温 | 「小白：体温下降，注意观察」 | 录入体温低于阈值时触发 |
| 生产记录 | 「小白第3窝：首次驱虫」 | 出生日期 + 14天 |
| 生产记录 | 「小白第3窝：确认断奶」 | 出生日期 + 45天（默认，可在 families 设置中配置） |
| 生产记录 | 「小白第3窝：首次疫苗」 | 出生日期 + 42天（可配置） |
| 疫苗记录 | 「幼崽A：第二针疫苗」 | 上一针 + 21天（可配置） |
| 驱虫记录（幼犬） | 「幼崽A：下次驱虫」 | 上一次 + 14天（可配置） |
| 驱虫记录（成犬） | 「种母小白：下次驱虫」 | 上一次 + 可配置周期 |
| 成犬疫苗 | 「种母小白：年度疫苗」 | 上一次 + 365天 |
| 历史发情 | 「小白：预计下次发情」 | 基于历史发情间隔推算 |
| 连续用药 | 「杠上花：黄体酮 第X天/共15天」 | 每日提醒直到完成 |
| 护理群组 | 「孕期护理(3只)：每日喂钙铁」 | 按状态群组规则 |

所有提醒的时间间隔都有系统默认值，但允许在录入时修改（不同疫苗、不同驱虫药间隔不同）。

> **默认断奶龄：** 默认断奶龄 45 天，可在 families.settings 中配置。

### 2.2 tasks 集合：预生成任务模型

> **核心设计：任务在写入时预生成（PRE-GENERATED），而非读取时计算。** 首页加载仅需对 tasks 集合执行一次查询。

**tasks 集合结构：**

```
{
  _id
  dog_id
  dog_name（冗余，避免联表查询）
  litter_id（选填）
  cycle_id（选填，关联繁育周期，用于周期关闭时批量取消该周期所有未完成任务）
  type: 'vaccination' | 'deworming' | 'breeding_milestone' | 'medication_daily' | 'care_group' | 'custom' | 'manual'
  // manual = 用户通过表单"标记为待办"开关手动创建的任务（task-service.createManualTask）
  title（任务标题，如「首次驱虫」「黄体酮 第3天/共15天」）
  due_date（timestamp 毫秒数）
  status: 'pending' | 'completed' | 'postponed' | 'cancelled'
  priority: 'overdue' | 'today' | 'upcoming'
  completed_by（选填，完成人 user_id）
  completed_at（选填，完成时间，timestamp 毫秒数）
  postpone_count（推迟次数，默认 0。连续推迟 3+ 次不处理 → 视觉升级回红色）
  postpone_reason（选填，最近一次推迟的原因）
  source_record_id（来源记录 ID，如 vaccination_id、mating_id）
  source_collection（来源集合名，如 'health_records'、'breeding_records'）
  family_id
  created_at                        // timestamp 毫秒数
  updated_at                        // timestamp 毫秒数
  deleted_at                        // timestamp 毫秒数，默认 null
}
```

**首页查询示例：**

```javascript
// clientDB JQL 方式
db.collection('tasks')
  .where(`family_id=="${familyId}" && status=="pending" && due_date<=${endOfWeek} && deleted_at==null`)
  .orderBy('due_date', 'asc')
  .get()

// 云对象中使用传统 API
const dbCmd = db.command
db.collection('tasks')
  .where({
    family_id: familyId,
    status: 'pending',
    due_date: dbCmd.lte(endOfWeek),
    deleted_at: null
  })
  .orderBy('due_date', 'asc')
  .get()
```

**任务生成三层保障机制：**

1. **同步生成（写入时）：** 当创建繁育记录、健康记录、用药任务等时，在同一请求中同步生成对应的 tasks 文档。例如：创建配种记录 → 同步生成「预计孕检日」和「预产期」任务。
2. **写后验证（post-write verification）：** 写入完成后异步检查任务是否正确生成，补漏。
3. **每日审计（scheduled function）：** 每日定时任务扫描所有活跃记录，确认对应任务存在且状态正确，修复不一致。

#### 三层保障实现细节

**Layer 2: 写后验证（同步）**
- 在云对象的写操作完成后，同一请求内立即执行验证查询
- 查询：`db.collection('tasks').where({ source_record_id: newRecordId, source_collection: collectionName }).count()`
- 如果 count === 0，立即补生成 task
- 不使用异步/后台任务，确保用户看到操作成功时 task 一定已创建
- 超时策略：验证查询超时 3 秒，超时则跳过（依赖 Layer 3 兜底）

**Layer 3: 每日审计（定时云函数）**
- 触发时间：每日 02:00 UTC+8
- 审计范围：
  - breeding_records: type=mating 且 details.expected_due_date 存在 → 应有预产期提醒 task
  - breeding_records: type=mating 且 details.expected_checkup_date 存在 → 应有孕检提醒 task
  - health_records: type=vaccination 且 details.next_reminder_date 存在 → 应有疫苗提醒 task
  - health_records: type=deworming 且 details.next_reminder_date 存在 → 应有驱虫提醒 task
  - medication_tasks: status=active → 应有每日用药 task
  - **care_rules 审计（独立逻辑）：** 遍历 families.care_rules，对每条规则查询当前处于 status_trigger 状态的犬只，检查是否存在对应的 pending task（type=care_group）。护理任务无 source_record_id（模板驱动），审计通过 `dog_id + type=care_group + title 匹配规则描述` 来判断任务是否存在。
- 查询约束：只审计活跃记录（非终态状态）
- 幂等性保证：生成 task 前检查 `tasks.where({ source_record_id, source_collection }).count()`，已存在则跳过
- 错误处理：审计失败时记录错误日志到 `audit_logs` 集合，不发送用户通知
- 性能：30-50 只犬的数据量下，审计查询为毫秒级，无需分批

**任务取消：** 繁育周期关闭（失败/放弃）、犬只死亡等场景触发批量取消，status 设为 cancelled。取消的任务保留记录不删除，便于审计。

**任务归档：** 已完成任务（status=completed）超过 30 天的，通过定时任务清理（软删除或移至归档集合），保持 tasks 集合精简高效。

### 2.3 连续用药任务

支持「每天 × N天」的重复提醒：
- 录入时指定：药品、犬只（可多选）、频率、持续天数
- 首页卡片显示进度（第X天/共N天）
- 每天标记完成后自动推进到下一天
- 漏打的天数标红提醒
- 完成全部天数后任务自动结束

### 2.4 状态群组提醒

用户自定义规则，犬只按状态自动加入/退出：

> **CareGroupRule 嵌入在 families 文档中**（families.care_rules 数组），不作为独立集合。

规则：
- 犬进入对应状态 → 自动加入该群组的每日任务
- 犬退出对应状态 → 自动退出
- 首次使用时系统提供常见模板（孕期喂钙铁、幼犬定期驱虫等）
- V1 护理规则覆盖整个状态期间，不做分阶段（day_range 后续迭代）

### 2.5 推送通知

**角色过滤：**
- 管理员（创建者/管理员）：收到所有推送
- 协助者：只收到日常护理类推送（疫苗、驱虫、每日用药、喂养）
- 不收到：繁育决策类（配种/孕检/预产期）、财务类
- 可在设置中调整（关闭某类推送）

**晨间摘要推送：**
- 每日定时推送一条摘要（默认早上7:00，可配置）
- 内容：「今日待办(7件)：2件逾期需处理 · 花花预产期倒计时5天 · 豆豆第3窝3只待驱虫」
- 一天一次，不打扰
- 点击通知直接进入首页

## 三、自动状态系统

### 3.1 自动推导的状态

> **状态为实时计算（REAL-TIME COMPUTED），不存储在 dog 文档上。** 每次查询时从 breeding_cycles、health_records、medication_tasks 等集合实时推导。在 30-50 只犬的规模下，此计算为毫秒级，无需缓存。

| 状态 | 计算依据 | 退出条件 |
|------|---------|---------|
| 发情中 | breeding_records 中有 type=heat 的记录且 details.end_date 为空 | 配种(→怀孕中) / 手动关闭 / 21天自动关闭 |
| 怀孕中 | breeding_records 中有 type=mating 的记录，且对应 breeding_cycle 状态为怀孕中 | 生产(→哺乳中) / 确认未怀孕/流产(→正常) |
| 哺乳中 | litters 中有该犬（dam_id）的记录且 weaned_at == null | 用户确认断奶（填入 weaned_at）/ 幼崽到达默认断奶龄时提醒确认 |
| 生病中 | health_records 中有 type=illness 的记录且 details.treatment_status ≠ 已康复 | 标记康复 |
| 用药中 | medication_tasks 中有 status=进行中 的记录 | 用药完成 / 取消 |
| 正常 | 以上都不满足 | — |

**繁育状态互斥：** 发情中/怀孕中/哺乳中同时只能有一个

**健康状态可叠加：** 生病中、用药中可与繁育状态并存

**多状态时 UI 展示：** 犬只卡片显示所有活跃状态标签

### 3.2 状态驱动护理群组

状态变化时自动加入/退出对应的护理群组：
- 配种后进入「怀孕中」→ 自动加入「孕期护理」群组
- 确认未怀孕 → 自动退出
- 标记生病 → 如有对应群组规则则加入
- 退休 → 不影响健康类群组，但不会进入繁育类群组

### 3.3 完整状态流转图

```
正常(无标签)
  ↓ 录入发情记录
发情中
  ├→ 配种 → 怀孕中
  │          ├→ 生产 → 哺乳中 → 确认断奶 → 正常
  │          └→ 流产/确认未怀孕/医疗终止 → 正常
  └→ 未配种(手动/21天自动) → 正常

健康状态（可与繁育状态并存）：
  正常 → 标记生病/快速标记 → 生病中 → 标记康复 → 正常
  任何状态 → 创建连续用药任务 → 用药中 → 用药完成 → 退出

终态：
  任何状态 → 标记死亡 → 已故（终态）
  种狗在养 → 标记退休 → 已退休（可逆）
```

### 3.4 快速状态标记（紧急情况）

犬只突然生病时，用户可能来不及创建完整的疾病记录。提供快速标记入口：

**入口位置：** 犬只档案详情页，顶部状态标签旁的「标记状态」按钮。

**快速标记流程：**
1. 点击「标记状态」→ 弹出选项：生病 / 其他
2. 选择「生病」→ 选择病症类型（感冒/腹泻/皮肤病/其他）
3. 确认 → 犬只立即进入「生病中」状态，加入对应护理群组
4. 系统自动生成一条 task：「请补录 XX 的完整疾病记录」，提醒用户后续补充详细信息（治疗方案、用药等）

**协助者也可使用：** 饲养员发现犬只异常时，可快速标记，管理员稍后补录完整记录。

**与完整记录的关系：** 快速标记创建一条简化的 health_record（type=illness，只有 dog_id + condition + start_date），后续补录时更新这条记录的详细字段。

## 四、协作权限

### 4.1 三级角色

| 角色 | 数量 | 权限 |
|------|------|------|
| 创建者 | 1个，不可转让 | 全部权限 + 删除家庭 + 移除管理员 |
| 管理员 | 不限 | 全部权限（日常体验与创建者相同） |
| 协助者 | 不限 | 查看任务、标记完成、添加备注、执行批量操作 |

**协助者不能访问的功能：**
- 查看/操作财务数据
- 销售管理
- 创建/编辑繁育记录（配种决策等）
- 系统设置/提醒规则
- 邀请/移除成员

**协助者可以做的：**
- 查看每日任务（首页卡片）
- 标记任务完成（打了疫苗、喂了药等）
- 执行批量操作（批量驱虫、批量喂药）
- 查看犬只档案
- 添加简单备注（如「某只狗今天没吃东西」）
- 快速标记犬只状态（标记生病，紧急情况下）

**推迟逾期任务：** 协助者可以推迟逾期任务（选择新日期+填原因），因为在只有协助者在场时需要灵活处理。

### 4.2 邀请机制

- 创建者/管理员生成邀请链接
- 被邀请人点击链接注册加入
- 默认角色：协助者
- 创建者/管理员可后续调整角色

> **uni-id-pages 预置登录 UI：** 可直接使用 uni-id-pages 插件提供的登录/注册/密码重置等完整页面，无需从零开发。被邀请人点击链接后进入 uni-id-pages 的注册页面完成注册，然后通过自定义云对象加入家庭。

### 4.3 操作归因

- 所有任务完成记录显示操作人和时间：「小李已完成 · 14:30」
- 替代微信群里的打卡汇报
- 创建者/管理员可查看操作日志

> **V1 限制：单家庭模式。** 一个用户只能属于一个家庭。云对象 `_before` 拦截器基于此假设直接从用户信息注入 `familyId`，无需用户选择。未来支持多家庭需重构 auth 拦截器和用户-家庭关联逻辑。

## 六、决策日志

| 决策 | 选择 | 考虑的替代方案 | 选择原因 |
|------|------|-------------|---------|
| 犬只分类方式 | 两个独立字段（性别+角色）+ 筛选 | 按性别分层级导航、按角色分层级导航 | 更灵活，支持任意组合筛选 |
| 首页主要展示 | 任务列表优先 + 日期条辅助 | 日历优先 + 列表在下 | 5个智能体一致投票：用户需求是执行不是规划 |
| 卡片组织方式 | 注意力单元（犬/群组/批量混合） | 纯以犬为主 / 纯以事件为主 | 匹配繁育者真实思维模式（来自微信数据分析） |
| 配种后状态 | 直接设为「怀孕中」 | 三段状态（已配种→待孕检→怀孕中） | 用户实际按怀孕护理，孕检可选非必须 |
| 费用存储 | 收支分表 + 繁育记录费用进Expense表 | 收支合并一表 / 繁育费用存记录上 | 统计简单（一张Expense表），语义清晰（BreedingCycle和Litter分开） |
| BreedingCycle 和 Litter | 分开（中间方案） | 合并为一个实体 | 语义最清晰，失败繁育不留空壳，费用可区分阶段 |
| 多犬费用分摊 | 动态计算（总额/犬数） | 存储分摊金额 | 更简单，修改总额自动生效 |
| 底部导航第四个 | 💰财务 | 📅日历 | 财务是核心痛点之一，日历功能由首页7天条覆盖 |
| 退休犬去向 | 领养（简单记录）/ 赠送 | 走正常销售流程 | 退休犬领养是小偿行为，不需要复杂流程 |
| 同一犬出现在多张卡片 | 允许（方案A） | 严格去重 | 个体任务和批量任务是不同的工作场景 |
| 销售记录结构 | 单表所有字段 | 独立收款流水表 | 实际场景简单（一笔定金+一笔尾款），不需要复杂流水 |
| 逾期项处理 | 推迟需选新日期+可填原因 | 红色警告置顶不可操作 | 理解用户处境（可能在等兽医），不是审判 |
| 收入数据管理 | 独立 Income 集合 | 从 SaleRecord 聚合收入 / 不建收入表 | 职责分离：SaleRecord 管流程，Income 管钱，Expense 管支出 |
| 日期统一方案 | timestamp 毫秒数（Number） | Date 对象 / YYYY-MM-DD 字符串 | UniCloud $env:now 返回毫秒数，与生态一致；避免时区问题 |
| disposition 变更信息 | disposition_date + disposition_notes 两个顶层字段 | 三个独立嵌入对象(death_info/adoption_info/gift_info) / 一个统一嵌入对象 | 字段少（仅2个），金额由 Income 表管理，死因/领养说明写在 notes 中足够 |
| 首页任务模型 | tasks 集合预生成（写入时生成） | 读取时实时计算 | 首页高频访问，预生成让加载只需 1 次查询 |
| 逾期任务展示 | 混入今日列表顶部（Apple Reminders模式） | 独立"需处理"分区（TickTick模式） | 30-50犬规模不需要分区，一个列表更直观，消除重复显示 |
| WeekStrip 布局 | 周一到周日固定排列，过去日期灰色不可点 | 以今天为中心（前3+今+后3） | 保持周感，消除过去日期空状态问题 |
| 摘要 pill 设计 | 今日/本周/30天（累进包含） | 需处理/今日/本周 | 逾期并入今日后不需要独立 pill，30天比本月更稳定（无月边界跳跃） |
| 犬只状态存储 | 实时从源数据推导，不存储 | 存储在 dog 文档的 current_statuses 数组 | 30-50 犬规模下实时计算毫秒级，预存引入一致性风险不值得 |
