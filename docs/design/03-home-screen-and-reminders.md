## 五、首页设计

### 5.1 三层结构

```
┌─ Zone 1：状态摘要栏（固定，不滚动）────────────────┐
│  [🔴 需处理 X]    [🟡 今日 X]    [本周 X]            │
│  点击数字 → 滚动到卡片区对应分类并高亮                │
├─ 7天预览条（固定）──────────────────────────────────┤
│  今  明  周三  周四  周五  周六  周日                  │
│  7   3    0    5    1     4    0                     │
│  点击某天 → 下方卡片区切换为该天事项                   │
│  点击月份标题 → 展开完整月历（用于规划）               │
├─ Zone 2：犬只卡片区（可滚动）───────────────────────┤
│  智能卡片按优先级分区：逾期 → 今日 → 即将到来          │
└─────────────────────────────────────────────────────┘
```

Zone 1 数字说明：
- 「需处理」= 逾期事项数量
- 「今日」= 今日待办事项数量（任务数，不是卡片数）
- 「本周」= 未来7天事项数量
- 数字可点击，滚动到对应区域

7天预览条：
- 显示每天事件数量
- 左右滑动查看更多日期
- 约 40px 高度，不占主要空间
- 点击月份标题展开完整月历

### 5.2 智能卡片系统

**卡片组织原则：以「注意力单元」为主，系统自动匹配最合适的展示方式。**

不是纯粹以犬为主，也不是以事件为主，而是匹配繁育者实际的思维方式：
- 照顾某只犬时 → 以犬为中心
- 给全场犬驱虫时 → 以事件为中心
- 给所有怀孕犬喂钙铁时 → 以状态群组为中心

**四种卡片类型：**

**① 个体犬只卡片**

某只犬有独立任务（配种/孕检/卵泡检查等）

```
┌──────────────────────────────────┐
│  大米 · 发情第14天                 │
│  今日打卵泡（已预约）              │
│  [记录结果]                       │
└──────────────────────────────────┘
```

**② 状态群组卡片**

某个状态群组有共同任务

```
┌──────────────────────────────────┐
│  🤰 孕期护理 · 杠上花、安妮、小乔  │
│  每日喂钙铁                       │
│  杠上花：怀孕第3天                 │
│  安妮：怀孕第18天                  │
│  小乔：怀孕第45天                  │
│  [批量标记今日已完成]              │
└──────────────────────────────────┘
```

**③ 批量卡片**

同类事件 + 同天 + 3只以上犬

```
┌──────────────────────────────────┐
│  💉 月度驱虫 · 今日 · 12只成犬     │
│  小白、花花、豆豆、妞妞 等12只     │
│  [批量处理全部]  [逐只查看]        │
└──────────────────────────────────┘
```

全部犬只时：「月度驱虫 · 今日 · 全部成犬(35只)」

**④ 连续用药卡片**

```
┌──────────────────────────────────┐
│  💊 每日用药 · 第3天/共15天        │
│  杠上花 · 黄体酮                   │
│  李拜六、莉莉娅 · 头孢曲松钠       │
│  [批量标记今日已完成]              │
└──────────────────────────────────┘
```

**卡片合并规则：**

| 条件 | 展示方式 |
|------|---------|
| 同类事件 + 同天 + 1-2只犬 | 各自独立的犬只卡片 |
| 同类事件 + 同天 + 3只以上 | 合并为一张批量卡片 |
| 同窝幼崽的同类事件 | 合并为一张窝卡片 |
| 同一只犬的多个任务 | 合并为一张犬只卡片，按优先级排任务 |
| 同一只犬既有个体任务又属于批量/群组任务 | 允许出现两次（方案A） |

**合并优先级（从高到低）：**

1. **窝级别合并**（最高优先）：同窝幼崽的同类事件 → 合并为窝卡片
2. **批量合并**：非同窝、同类事件 + 同天 + 3只以上 → 合并为批量卡片
3. **个体卡片**：不满足以上条件的 → 各自独立卡片

**冲突场景示例：**
- 花花第3窝5只幼崽今天都需要打疫苗 → 按规则1，合并为「窝卡片」（不走规则2的批量卡片）
- 3只不同窝的成犬今天都需要驱虫 → 按规则2，合并为「批量卡片」
- 花花第3窝3只 + 豆豆第2窝2只今天都需要疫苗 → 各窝分别合并为窝卡片（两张卡片），不跨窝合并

**单卡任务数限制：**
- 单张犬只卡片最多显示 3 个任务，超出显示「还有X项」
- 批量卡片最多显示 12 只犬，超出分页或显示「还有X只」

**一只犬只有多任务时，只出现一张卡片，放在最高优先级区域：**

```
🔴 需立即处理
┌──────────────────────────────────┐
│  小白 · 马尔济斯 · 2岁            │
│  🔴 疫苗逾期 2天                  │
│  🟡 今日体检                      │
│  [去处理疫苗]  [记录体检]          │
└──────────────────────────────────┘
```

### 5.3 卡片核心设计规则

**以犬为主语，不是以任务为主语：**
- 「花花 · 孕期第58天」而非「孕检任务」
- 同一只犬的多个任务合并为一张卡片
- 同一窝的批量任务合并（「豆豆第3窝 5只幼崽 · 首次驱虫」）

**状态语言优于日期语言：**
- 「孕期第58天 · 距预产期5天」而非「3月26日预产」
- 「发情第12天 · 配种窗口期」而非「3月21日建议配种」

**操作零跳转——卡片内嵌行动按钮：**
- 简单操作（标记完成、记录单次事件）→ 底部弹出表单（Bottom Sheet），不离开首页
- 复杂操作（批量处理、分娩记录）→ 跳转专属页面，完成后返回首页
- 所有操作完成后有明确视觉反馈（卡片消失/变灰 + Zone 1 数字更新）
- 触控目标 ≥ 44pt（适配脏手/湿手场景）

**逾期是提醒，不是审判：**
- 逾期卡片提供「推迟并说明原因」选项
- 推迟时必须选择新的提醒日期（默认明天）
- 可选填原因（如「等兽医回复」）
- 推迟后移出逾期区域，到新日期再出现
- 连续推迟超过3次不处理，视觉再次升级为红色

**批量任务显示进度：**

```
┌──────────────────────────────────┐
│  豆豆第3窝 · 首次驱虫 · 3/5只待处理│
│  ✅ 幼崽A ✅ 幼崽B                │
│  ⬜ 幼崽C  ⬜ 幼崽D  ⬜ 幼崽E     │
│  [批量处理剩余3只]                │
└──────────────────────────────────┘
```

**协作透明：**
- 已完成任务显示操作人和时间：「小李已完成 · 14:30」

**卡片数量控制：**
- 每个优先级区域最多显示 6-8 张卡片
- 超出部分折叠为「还有 X 件事项 ▾」
- 0 件事时：显示「犬群一切正常 · 距下一个事项还有 X 天」

### 5.4 底部导航

```
🏠首页  🐾档案  ➕快速记录  💰财务  👤我的
```

**中间 "+" FAB 行为：** 点击 → 弹出 FAB Action Sheet（智能推荐 3 个 + 常用 4 个 + 「全部记录类型 →」链接）。R-1 完整记录类型页面降级为二级入口，从 Action Sheet 底部「全部记录类型」进入。

## 六、提醒/通知系统

### 6.2 tasks 集合：预生成任务模型

> **核心设计：任务在写入时预生成（PRE-GENERATED），而非读取时计算。** 首页加载仅需对 tasks 集合执行一次查询。

**tasks 集合结构：**

```
{
  _id
  dog_id
  dog_name（冗余，避免联表查询）
  litter_id（选填）
  cycle_id（选填，关联繁育周期，用于周期关闭时批量取消该周期所有未完成任务）
  type: 'vaccination' | 'deworming' | 'breeding_milestone' | 'medication_daily' | 'care_group' | 'custom'
  title（任务标题，如「首次驱虫」「黄体酮 第3天/共15天」）
  due_date（timestamp 毫秒数）
  status: 'pending' | 'completed' | 'postponed' | 'cancelled'
  priority: 'overdue' | 'today' | 'upcoming'
  completed_by（选填，完成人 user_id）
  completed_at（选填，完成时间，timestamp 毫秒数）
  postpone_count（推迟次数，默认 0。连续推迟 3+ 次不处理 → 视觉升级回红色）
  postpone_reason（选填，最近一次推迟的原因）
  source_record_id（来源记录 ID，如 vaccination_id、mating_id）
  source_collection（来源集合名，如 'health_records'、'breeding_records'）
  family_id
  created_at                        // timestamp 毫秒数
  updated_at                        // timestamp 毫秒数
  deleted_at                        // timestamp 毫秒数，默认 null
}
```

**首页查询示例：**

```javascript
// clientDB JQL 方式
db.collection('tasks')
  .where(`family_id=="${familyId}" && status=="pending" && due_date<=${endOfWeek} && deleted_at==null`)
  .orderBy('due_date', 'asc')
  .get()

// 云对象中使用传统 API
const dbCmd = db.command
db.collection('tasks')
  .where({
    family_id: familyId,
    status: 'pending',
    due_date: dbCmd.lte(endOfWeek),
    deleted_at: null
  })
  .orderBy('due_date', 'asc')
  .get()
```

**任务生成三层保障机制：**

1. **同步生成（写入时）：** 当创建繁育记录、健康记录、用药任务等时，在同一请求中同步生成对应的 tasks 文档。例如：创建配种记录 → 同步生成「预计孕检日」和「预产期」任务。
2. **写后验证（post-write verification）：** 写入完成后异步检查任务是否正确生成，补漏。
3. **每日审计（scheduled function）：** 每日定时任务扫描所有活跃记录，确认对应任务存在且状态正确，修复不一致。

#### 三层保障实现细节

**Layer 2: 写后验证（同步）**
- 在云对象的写操作完成后，同一请求内立即执行验证查询
- 查询：`db.collection('tasks').where({ source_record_id: newRecordId, source_collection: collectionName }).count()`
- 如果 count === 0，立即补生成 task
- 不使用异步/后台任务，确保用户看到操作成功时 task 一定已创建
- 超时策略：验证查询超时 3 秒，超时则跳过（依赖 Layer 3 兜底）

**Layer 3: 每日审计（定时云函数）**
- 触发时间：每日 02:00 UTC+8
- 审计范围：
  - breeding_records: type=mating 且 details.expected_due_date 存在 → 应有预产期提醒 task
  - breeding_records: type=mating 且 details.expected_checkup_date 存在 → 应有孕检提醒 task
  - health_records: type=vaccination 且 details.next_reminder_date 存在 → 应有疫苗提醒 task
  - health_records: type=deworming 且 details.next_reminder_date 存在 → 应有驱虫提醒 task
  - medication_tasks: status=active → 应有每日用药 task
  - **care_rules 审计（独立逻辑）：** 遍历 families.care_rules，对每条规则查询当前处于 status_trigger 状态的犬只，检查是否存在对应的 pending task（type=care_group）。护理任务无 source_record_id（模板驱动），审计通过 `dog_id + type=care_group + title 匹配规则描述` 来判断任务是否存在。
- 查询约束：只审计活跃记录（非终态状态）
- 幂等性保证：生成 task 前检查 `tasks.where({ source_record_id, source_collection }).count()`，已存在则跳过
- 错误处理：审计失败时记录错误日志到 `audit_logs` 集合，不发送用户通知
- 性能：30-50 只犬的数据量下，审计查询为毫秒级，无需分批

**任务取消：** 繁育周期关闭（失败/放弃）、犬只死亡等场景触发批量取消，status 设为 cancelled。取消的任务保留记录不删除，便于审计。

**任务归档：** 已完成任务（status=completed）超过 30 天的，通过定时任务清理（软删除或移至归档集合），保持 tasks 集合精简高效。

### 6.3 连续用药任务

支持「每天 × N天」的重复提醒：
- 录入时指定：药品、犬只（可多选）、频率、持续天数
- 首页卡片显示进度（第X天/共N天）
- 每天标记完成后自动推进到下一天
- 漏打的天数标红提醒
- 完成全部天数后任务自动结束

### 6.4 状态群组提醒

用户自定义规则，犬只按状态自动加入/退出：

> **CareGroupRule 嵌入在 families 文档中**（families.care_rules 数组），不作为独立集合。

**families.care_rules 数组元素结构：**

```
{
  status_trigger: 怀孕中/哺乳中/生病中/用药中/...
  task_description（任务描述，如「每日喂钙铁」）
  frequency: 每日/每周/自定义
}
```

规则：
- 犬进入对应状态 → 自动加入该群组的每日任务
- 犬退出对应状态 → 自动退出
- 首次使用时系统提供常见模板（孕期喂钙铁、幼犬定期驱虫等）
- V1 护理规则覆盖整个状态期间，不做分阶段（day_range 后续迭代）

### 6.5 推送通知

**角色过滤：**
- 管理员（创建者/管理员）：收到所有推送
- 协助者：只收到日常护理类推送（疫苗、驱虫、每日用药、喂养）
- 不收到：繁育决策类（配种/孕检/预产期）、财务类
- 可在设置中调整（关闭某类推送）

**晨间摘要推送：**
- 每日定时推送一条摘要（默认早上7:00，可配置）
- 内容：「今日待办(7件)：2件逾期需处理 · 花花预产期倒计时5天 · 豆豆第3窝3只待驱虫」
- 一天一次，不打扰
- 点击通知直接进入首页

## 七、自动状态系统

### 7.1 自动推导的状态

> **状态为实时计算（REAL-TIME COMPUTED），不存储在 dog 文档上。** 每次查询时从 breeding_cycles、health_records、medication_tasks 等集合实时推导。在 30-50 只犬的规模下，此计算为毫秒级，无需缓存。

| 状态 | 计算依据 | 退出条件 |
|------|---------|---------|
| 发情中 | breeding_records 中有 type=heat 的记录且 details.end_date 为空 | 配种(→怀孕中) / 手动关闭 / 21天自动关闭 |
| 怀孕中 | breeding_records 中有 type=mating 的记录，且对应 breeding_cycle 状态为怀孕中 | 生产(→哺乳中) / 确认未怀孕/流产(→正常) |
| 哺乳中 | litters 中有该犬（dam_id）的记录且 weaned_at == null | 用户确认断奶（填入 weaned_at）/ 幼崽到达默认断奶龄时提醒确认 |
| 生病中 | health_records 中有 type=illness 的记录且 details.treatment_status ≠ 已康复 | 标记康复 |
| 用药中 | medication_tasks 中有 status=进行中 的记录 | 用药完成 / 取消 |
| 正常 | 以上都不满足 | — |

**繁育状态互斥：** 发情中/怀孕中/哺乳中同时只能有一个

**健康状态可叠加：** 生病中、用药中可与繁育状态并存

**多状态时 UI 展示：** 犬只卡片显示所有活跃状态标签

### 7.2 状态驱动护理群组

状态变化时自动加入/退出对应的护理群组：
- 配种后进入「怀孕中」→ 自动加入「孕期护理」群组
- 确认未怀孕 → 自动退出
- 标记生病 → 如有对应群组规则则加入
- 退休 → 不影响健康类群组，但不会进入繁育类群组

### 7.3 完整状态流转图

```
正常(无标签)
  ↓ 录入发情记录
发情中
  ├→ 配种 → 怀孕中
  │          ├→ 生产 → 哺乳中 → 确认断奶 → 正常
  │          └→ 流产/确认未怀孕/医疗终止 → 正常
  └→ 未配种(手动/21天自动) → 正常

健康状态（可与繁育状态并存）：
  正常 → 标记生病/快速标记 → 生病中 → 标记康复 → 正常
  任何状态 → 创建连续用药任务 → 用药中 → 用药完成 → 退出

终态：
  任何状态 → 标记死亡 → 已故（终态）
  种狗在养 → 标记退休 → 已退休（可逆）
```

#### 7.4 快速状态标记（紧急情况）

犬只突然生病时，用户可能来不及创建完整的疾病记录。提供快速标记入口：

**入口位置：** 犬只档案详情页，顶部状态标签旁的「标记状态」按钮。

**快速标记流程：**
1. 点击「标记状态」→ 弹出选项：生病 / 其他
2. 选择「生病」→ 选择病症类型（感冒/腹泻/皮肤病/其他）
3. 确认 → 犬只立即进入「生病中」状态，加入对应护理群组
4. 系统自动生成一条 task：「请补录 XX 的完整疾病记录」，提醒用户后续补充详细信息（治疗方案、用药等）

**协助者也可使用：** 饲养员发现犬只异常时，可快速标记，管理员稍后补录完整记录。

**与完整记录的关系：** 快速标记创建一条简化的 health_record（type=illness，只有 dog_id + condition + start_date），后续补录时更新这条记录的详细字段。
