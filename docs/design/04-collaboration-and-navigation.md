### 八、协作权限

> **MongoDB 适配说明：** 支持软删除的集合使用 deleted_at 字段（见 01 文档软删除范围矩阵）。

#### 8.1 三级角色

| 角色 | 数量 | 权限 |
|------|------|------|
| 创建者 | 1个，不可转让 | 全部权限 + 删除家庭 + 移除管理员 |
| 管理员 | 不限 | 全部权限（日常体验与创建者相同） |
| 协助者 | 不限 | 查看任务、标记完成、添加备注、执行批量操作 |

**协助者不能访问的功能：**
- 查看/操作财务数据
- 销售管理
- 创建/编辑繁育记录（配种决策等）
- 系统设置/提醒规则
- 邀请/移除成员

**协助者可以做的：**
- 查看每日任务（首页卡片）
- 标记任务完成（打了疫苗、喂了药等）
- 执行批量操作（批量驱虫、批量喂药）
- 查看犬只档案
- 添加简单备注（如「某只狗今天没吃东西」）
- 快速标记犬只状态（标记生病，紧急情况下）

**推迟逾期任务：** 协助者可以推迟逾期任务（选择新日期+填原因），因为在只有协助者在场时需要灵活处理。

#### 8.2 邀请机制

- 创建者/管理员生成邀请链接
- 被邀请人点击链接注册加入
- 默认角色：协助者
- 创建者/管理员可后续调整角色

> **uni-id-pages 预置登录 UI：** 可直接使用 uni-id-pages 插件提供的登录/注册/密码重置等完整页面，无需从零开发。被邀请人点击链接后进入 uni-id-pages 的注册页面完成注册，然后通过自定义云对象加入家庭。

#### 8.3 操作归因

- 所有任务完成记录显示操作人和时间：「小李已完成 · 14:30」
- 替代微信群里的打卡汇报
- 创建者/管理员可查看操作日志

#### 8.4 Family / User 数据模型

> **V1 限制：单家庭模式。** 一个用户只能属于一个家庭。云对象 `_before` 拦截器基于此假设直接从用户信息注入 `familyId`，无需用户选择。未来支持多家庭需重构 auth 拦截器和用户-家庭关联逻辑。

> **MongoDB 适配：** Family 和 FamilyMember 合并为单个 families 文档，成员和护理规则作为嵌入数组存储。不再使用独立的 FamilyMember 表。

**families 文档结构：**

```
{
  _id
  name（家庭/犬舍名称）
  creator_id → User
  created_at                       // timestamp 毫秒数
  updated_at                       // timestamp 毫秒数
  members: [
    {
      user_id → User
      role: 'creator' | 'admin' | 'helper'
      status: 'active' | 'invited' | 'removed'
      joined_at
      removed_at（选填）
    }
  ]
  care_rules: [
    {
      status_trigger（如：怀孕中/哺乳中/生病中/用药中）
      task_description（任务描述，如「每日喂钙铁」）
      frequency: 每日/每周/自定义
    }
  ]
  settings: {
    default_weaning_days（默认断奶龄，默认 45）
    default_vaccine_interval（默认疫苗间隔，默认 21）
    default_deworming_interval_puppy（幼犬驱虫间隔，默认 14）
    default_deworming_interval_adult（成犬驱虫间隔，默认 90）
    morning_summary_time（晨间摘要时间，默认 "07:00"）
  }
}
```

用户退出/移除后，其历史操作记录保留（completed_by 不受影响），对应 members 元素的 status 设为 'removed'。

### 九、导航与页面结构

#### 9.1 底部导航

```
🏠首页  🐾档案  ➕快速记录  💰财务  👤我的
```

#### 9.2 各页面功能

**🏠 首页**
- 三层结构：状态摘要栏 + 7天预览条 + 智能卡片区
- 核心功能：查看今日待办、处理任务、全局概览
- 月历：点击7天条的月份标题展开

**🐾 档案**
- 犬只列表：支持按性别、角色筛选
- 犬只详情页：
  - 基础信息
  - 当前状态标签
  - 繁育历史（按周期展示，每个周期展开可看子记录）
  - 健康记录（疫苗、驱虫、疾病、用药）
  - 财务摘要（该犬相关收支，可点击跳转财务页筛选视图）
  - 如果是种母：所有窝的列表
- 窝详情页（嵌套在种母繁育历史下）：
  - 窝基本信息（种公、生产日期、生产方式）
  - 幼崽列表（每只的状态、去向）
  - 窝利润统计
  - 「记录体重」按钮（批量体重录入入口）
  - 「+ 添加幼崽」按钮
- 犬只详情页需要提供「+ 添加记录」快捷入口（浮动按钮或顶部按钮）

**➕ 快速记录（FAB Action Sheet）**
- 点击底部导航中间 "+" FAB → 弹出 Action Sheet（非全页面跳转）
- Action Sheet 内容：
  - 智能推荐区（3个）：基于当前犬群状态智能推荐最可能需要的记录类型
  - 常用记录区（4个）：用户高频使用的记录类型
  - 底部链接：「全部记录类型 →」→ 跳转 R-1 完整记录类型页面
- R-1 快速记录入口页降级为二级入口（从 Action Sheet「全部记录类型」进入）
- R-1 完整页面包含所有 15 种记录类型的图标网格：
  - 繁育记录（发情/卵泡检查/配种/孕检/产检/临产监测/生产）
  - 健康记录（疫苗/驱虫/疾病/用药）
  - 财务记录（支出/收入）
  - 体重记录
  - 自定义记录
- 生产记录有专属的步骤式向导：
  步骤1：母犬信息（自动填充）+ 生产日期 + 生产方式 + 费用
  步骤2：逐只录入幼崽（性别、体重、存活状态、标识）[+ 添加下一只]
  步骤3：确认创建窝 + 经验心得
- 用药记录入口提供「从方案库选择」选项
- 用药记录完成后提示「是否保存为常用方案？」

**💰 财务**
- 统一流水列表（合并 Income + Expense 两表展示，按时间排序，收入红色/支出绿色；中式记账：红色=进账/吉利，绿色=出账）
- 筛选：按时间/按分类/按犬只/按窝
- 统计视图：月度/年度总账、单窝利润、单只种母ROI
- 支出录入入口
- 未来预估（数据积累后开启）

**👤 我的**
- 个人信息、家庭管理、成员邀请
- 通知设置（推送开关、晨间摘要时间）
- 用药方案库管理
- 合作代理人列表
- 支出分类自定义
- 数据备份/导出
- 设置

### 十、决策日志

| 决策 | 选择 | 考虑的替代方案 | 选择原因 |
|------|------|-------------|---------|
| 犬只分类方式 | 两个独立字段（性别+角色）+ 筛选 | 按性别分层级导航、按角色分层级导航 | 更灵活，支持任意组合筛选 |
| 首页主要展示 | 任务列表优先 + 日期条辅助 | 日历优先 + 列表在下 | 5个智能体一致投票：用户需求是执行不是规划 |
| 卡片组织方式 | 注意力单元（犬/群组/批量混合） | 纯以犬为主 / 纯以事件为主 | 匹配繁育者真实思维模式（来自微信数据分析） |
| 配种后状态 | 直接设为「怀孕中」 | 三段状态（已配种→待孕检→怀孕中） | 用户实际按怀孕护理，孕检可选非必须 |
| 费用存储 | 收支分表 + 繁育记录费用进Expense表 | 收支合并一表 / 繁育费用存记录上 | 统计简单（一张Expense表），语义清晰（BreedingCycle和Litter分开） |
| BreedingCycle 和 Litter | 分开（中间方案） | 合并为一个实体 | 语义最清晰，失败繁育不留空壳，费用可区分阶段 |
| 多犬费用分摊 | 动态计算（总额/犬数） | 存储分摊金额 | 更简单，修改总额自动生效 |
| 底部导航第四个 | 💰财务 | 📅日历 | 财务是核心痛点之一，日历功能由首页7天条覆盖 |
| 退休犬去向 | 领养（简单记录）/ 赠送 | 走正常销售流程 | 退休犬领养是小偿行为，不需要复杂流程 |
| 同一犬出现在多张卡片 | 允许（方案A） | 严格去重 | 个体任务和批量任务是不同的工作场景 |
| 销售记录结构 | 单表所有字段 | 独立收款流水表 | 实际场景简单（一笔定金+一笔尾款），不需要复杂流水 |
| 逾期项处理 | 推迟需选新日期+可填原因 | 红色警告置顶不可操作 | 理解用户处境（可能在等兽医），不是审判 |
| 收入数据管理 | 独立 Income 集合 | 从 SaleRecord 聚合收入 / 不建收入表 | 职责分离：SaleRecord 管流程，Income 管钱，Expense 管支出 |
| 日期统一方案 | timestamp 毫秒数（Number） | Date 对象 / YYYY-MM-DD 字符串 | UniCloud $env:now 返回毫秒数，与生态一致；避免时区问题 |
| disposition 变更信息 | disposition_date + disposition_notes 两个顶层字段 | 三个独立嵌入对象(death_info/adoption_info/gift_info) / 一个统一嵌入对象 | 字段少（仅2个），金额由 Income 表管理，死因/领养说明写在 notes 中足够 |
| 首页任务模型 | tasks 集合预生成（写入时生成） | 读取时实时计算 | 首页高频访问，预生成让加载只需 1 次查询 |
| 犬只状态存储 | 实时从源数据推导，不存储 | 存储在 dog 文档的 current_statuses 数组 | 30-50 犬规模下实时计算毫秒级，预存引入一致性风险不值得 |
