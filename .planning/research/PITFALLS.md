# Domain Pitfalls: Homepage Workbench Density Adaptation

**Project:** breed-app
**Domain:** existing dog breeding app homepage attention/workbench
**Researched:** 2026-04-15
**Overall confidence:** HIGH

## Roadmap Phase Guardrails

Recommended phase order for the density/workbench milestone:

| Phase | Name | Primary Risk It Controls |
|-------|------|--------------------------|
| Phase 1 | Contract Freeze + Test Harness | Prevent semantic regressions before changing density UI |
| Phase 2 | Backend Workbench Grouping Protocol | Keep card grouping, counts, and routing authoritative in `task-service` |
| Phase 3 | Frontend Density UI + Local Reconciliation | Add compact rows/folding without breaking optimistic removal and WeekStrip |
| Phase 4 | Scenario UAT + Performance Hardening | Validate real 30-50 dog workloads, partial completion, and race conditions |

Phase 1 should happen before any visual refactor. The current homepage behavior depends on contracts spread across `task-service`, `home/index.vue`, smart cards, health forms, and breeding record pages. A density UI change that skips contract tests is likely to pass `type-check` while shipping behavior regressions.

## Critical Pitfalls

### Pitfall 1: Reintroducing a Unified Task Pool

**What goes wrong:** Density work turns the homepage into a generic compact list where `逾期 / 繁育 / 健康 / 用药` become presentation filters instead of hard semantic sections.

**Why it happens:** Compact workbench layouts naturally invite a single "task row" abstraction. In this app, that abstraction is wrong because breeding workflow, health reminders, medication courses, and overdue handling have different completion semantics.

**Warning signs:**
- New code groups by `due_date` or `cardType` first, then assigns section labels later.
- `breeding_extra_arrangement` appears in health reminders.
- `sick_observation` appears in medication/therapy grouping.
- The top pills count rows generated by a new frontend grouping layer rather than backend `sectionType`.
- A new component accepts "any task" and exposes the same complete action for all domains.

**Prevention strategy:**
- Phase 1: freeze the section contract in tests: `workflow`, `workflow_extra`, `reminders`, `therapy`, plus overdue extraction.
- Phase 2: keep semantic classification in `task-service.buildSectionedCards`; frontend density logic should only choose expanded/card/row/folded rendering for already-classified cards.
- Phase 3: create section-specific renderers or adapters rather than one universal task row. A compact row may be reused visually, but commands must remain domain-specific.
- Phase 4: UAT with mixed data: one dog in breeding workflow, illness observation, and medication on the same day. It is valid for the same dog to appear in multiple sections.

**Phase should address:** Phase 1 and Phase 2.

### Pitfall 2: Making Breeding Workflow Batchable

**What goes wrong:** The breeding section is compacted by grouping tasks into a batch card with bulk complete/postpone semantics, treating flow steps like health tasks.

**Why it happens:** TODO-29 says "按流程步骤分组", which can be misread as "batch by same step". But breeding workflow is a state-machine推进器: a row may be grouped visually by step, yet each action must route to a form or litter page with locked context.

**Warning signs:**
- A breeding step group has a "完成全部" button.
- `breeding_milestone` enters the generic batch merge path.
- `breeding_milestone` cards lose `details.step_type`, `cycle_id`, `taskId`, or `locked=true`.
- `weaning_confirm` routes to a record form instead of `/pages/breeding/litter`.
- Completing an extra arrangement creates a `breeding_record` or changes cycle status.

**Prevention strategy:**
- Phase 1: add route contract tests or lightweight component tests for `follicle_check`, `pregnancy_check`, and `weaning_confirm`.
- Phase 2: backend may return `workflowGroups` by `step_type`, but group items must remain individual actionable rows with original `task` payload intact.
- Phase 3: breeding UI should use "step group + single dog/litter row" semantics. It may support group collapse/expand, not batch completion.
- Phase 4: UAT each route from homepage: follicle page, pregnancy page, litter detail, cycle detail for extra arrangements.

**Phase should address:** Phase 1, Phase 2, Phase 3.

### Pitfall 3: Breaking Health Batch Completion Source-of-Truth

**What goes wrong:** Dense health cards visually remove tasks or mark them completed without creating real `health_records`, or remove the whole batch card after only a subset completed.

**Why it happens:** Existing batch cards already do optimistic local removal before backend confirmation. Density adaptation may add new "bulk row" UI and accidentally use frontend selection state as truth.

**Warning signs:**
- `batchCompleteTask` is called with `autoRecord=false` for vaccination/deworming completion.
- Return payloads ignore backend `completedTasks` from health record pages.
- `removeBatchCard` is true whenever at least one task completed.
- Partial completion updates `tasks` but not `dogs`, `progress.total`, or title count.
- A compact health group allows "complete visible" when invisible/folded dogs are ambiguous.

**Prevention strategy:**
- Phase 1: add tests for "partial batch completion only removes completed subset" and "batch vaccination/deworming creates `health_records`".
- Phase 2: define a backend/frontend payload contract for density health groups: `sourceTaskIds`, `completedTasks`, `visibleTaskIds`, `groupKey`, and subtype fields.
- Phase 3: all local removal must consume authoritative completed IDs. If only some IDs complete, call `syncCardMeta`-equivalent logic for the new grouped structure.
- Phase 4: UAT three cases: all complete, one skipped, one backend failure. Verify dog files contain health records only for completed dogs.

**Phase should address:** Phase 1, Phase 2, Phase 3, Phase 4.

### Pitfall 4: Regressing WeekStrip Red Dots While Adding Density Counts

**What goes wrong:** Workbench counters, folds, or "今日重点" start using `counts.today` or compact group counts, causing WeekStrip red dots to disagree with visible cards.

**Why it happens:** The current code intentionally patches `dayCounts[todayTs]` after `Promise.all`, based on actual visible cards, because `counts.today` includes therapy semantics that are not equivalent to red-dot visibility.

**Warning signs:**
- `dayCounts` is written inside `loadTodayCards`, `loadDateCounts`, or `loadWeekCache`.
- Today red dot is derived from `counts.today`.
- Future dates with only `medication_tasks` lose red dots.
- Folded groups set day count to the number of groups, while clicking the date shows rows/cards not represented by that count.
- `loadAll` no longer uses a single `latestLoadToken` across all three requests.

**Prevention strategy:**
- Phase 1: preserve tests for medication-only future red dots and add one test for visible-card count vs `counts.today`.
- Phase 2: backend grouping protocol should return both display groups and raw card presence; red-dot logic should count day visibility, not row density.
- Phase 3: keep the today red-dot correction after `Promise.all`. If the UI introduces folded groups, use "has visible work" as the dot boolean, not expanded row count.
- Phase 4: manual race test: complete today's last medication dose, immediately switch dates, return today, and verify no stale red dot/card mismatch.

**Phase should address:** Phase 1 and Phase 3.

### Pitfall 5: Removing Latest-Token / Suppression During Refactor

**What goes wrong:** A task disappears after completion, then reappears when a slower background request resolves.

**Why it happens:** Density UI usually requires refactoring `cards`, `dayCards`, `weekCache`, and counts. The current protection is split across `latestLoadToken`, `suppressedTaskMap`, `filterSuppressedCards`, `removeCardLocally`, and `syncWeekCache`.

**Warning signs:**
- New load functions each own their own token.
- `loadAll` awaits requests sequentially or updates shared state after each response without token checks.
- Suppression only filters `cards`, not `weekCache`.
- Local removal is implemented separately for compact rows and bypasses `syncWeekCache`.
- Completed rows disappear from today but still show when selecting tomorrow/today from WeekStrip cache.

**Prevention strategy:**
- Phase 1: document and test the reconciliation contract with a small pure helper if possible.
- Phase 3: centralize density removal around one adapter: `removeTaskFromWorkbench(taskId, options)` that updates today, selected date, week cache, counts, card/group metadata, and suppression together.
- Phase 3: do not add per-component local deletion logic except ephemeral visual state; state mutation should remain page-level or store-level.
- Phase 4: simulate out-of-order responses: start `loadAll`, complete a task, then apply the stale response. The completed task must remain suppressed.

**Phase should address:** Phase 1, Phase 3, Phase 4.

### Pitfall 6: Medication Workbench Uses Old Task Model

**What goes wrong:** Medication density grouping uses legacy `tasks(type='medication')`, English statuses, or medication detail assumptions instead of `medication_tasks` with Chinese statuses and `daily_doses`.

**Why it happens:** The codebase still has compatibility paths and a known broken medication detail page. New grouping work may copy from older code instead of the current `MedicationCard` and `health-service` medication methods.

**Warning signs:**
- New code checks `status === 'active'` or `status === 'completed'` for medication_tasks.
- New UI reads `completed_dates`, `completed_map`, `start_date`, or `end_date`.
- Medication rows are generated from `tasks` rather than `medication_tasks`.
- A completed-for-today medication still displays as an urgent unfinished row.
- "已完成" dogs are mixed above unfinished dogs in the compact medication group.

**Prevention strategy:**
- Phase 1: add tests for medication grouping states: unfinished, partial, done today, multi-drug, sick_with_med, med_only.
- Phase 2: backend should return medication group metadata from `computeMedItemsForDay`, not a new frontend derivation.
- Phase 3: medication UI should sort by unfinished → partial → done, with done weak/folded, but commands still call `recordMedicationDose`, `batchCompleteMedicationDay`, and `endMedicationByDog`.
- Phase 4: UAT high-frequency dosage and multi-drug dogs; verify local dose optimism reconciles with backend refresh.

**Phase should address:** Phase 1, Phase 2, Phase 3.

### Pitfall 7: Density Fold Hides Actionable or Dangerous Items

**What goes wrong:** Compact mode folds away overdue, severe illness, imminent breeding steps, or unfinished medication behind "查看更多", making the homepage less useful than the current long list.

**Why it happens:** Generic thresholds such as "show first 3 rows per section" ignore domain priority. The product goal is density adaptation, not hiding work.

**Warning signs:**
- Fold logic sorts by creation order or dog name only.
- Overdue items are folded below non-overdue items.
- Severe illness observation is visually weaker than routine vaccine reminders.
- Done medication rows take space above unfinished rows.
- "今日重点" duplicates low-priority items while urgent items remain only in folded sections.

**Prevention strategy:**
- Phase 2: define priority scoring in backend or a shared helper: overdue first, breeding due/imminent, unfinished medication, severe illness, routine health reminders, completed/weak states.
- Phase 3: folded sections must expose counts by urgency, e.g. "还有 7 项，含 2 项逾期", not a bland total.
- Phase 3: "今日重点" should be derived from the same priority scoring and never replace the full section lists.
- Phase 4: UAT with 20 breeding, 20 health, 10 medication items and verify top viewport contains the right 3-5 attention units.

**Phase should address:** Phase 2, Phase 3, Phase 4.

## Moderate Pitfalls

### Pitfall 8: Backend Card Protocol Drift

**What goes wrong:** The backend adds density-specific shapes that do not preserve existing `SmartCard` fields, breaking `BFabSheet`, route generation, local removal, or cached rendering.

**Warning signs:**
- New grouped card lacks stable `id`, `cardType`, `domain`, `sectionType`, `priority`, or original `tasks`.
- Group rows do not include `_id`, `dog_id`, `dog_name`, `cycle_id`, `litter_id`, and `details`.
- `taskStore.cards` caches one shape while the page expects another.
- `SmartCardData` and cloud return shape diverge without a migration/adaptation layer.

**Prevention strategy:**
- Phase 1: write a card contract table before implementation.
- Phase 2: version the new protocol as additive fields: `displayMode`, `groups`, `rows`, `density`, while preserving original `tasks` and section metadata.
- Phase 3: adapt old cards to new workbench view in one helper, rather than changing every smart card consumer at once.

**Phase should address:** Phase 1 and Phase 2.

### Pitfall 9: Subtype Group Key Collisions

**What goes wrong:** Different vaccine types or deworming drug/type combinations collapse into one group, leading to wrong batch titles and wrong records.

**Warning signs:**
- Group keys use only `type + due_date`.
- Vaccination groups ignore `details.vaccine_type`.
- Deworming groups ignore `details.deworming_type` or `details.drug_name`.
- A batch page receives details from the first task and applies them to dogs with different subtype tasks.

**Prevention strategy:**
- Phase 1: keep tests for subtype-specific batch keys and add mixed subtype density tests.
- Phase 2: reuse `getTaskVariantKey` or move it to a tested helper before building new workbench groups.
- Phase 3: display group labels must include subtype/drug where relevant.

**Phase should address:** Phase 1 and Phase 2.

### Pitfall 10: Future Workflow Visibility Gets Lost

**What goes wrong:** Future `breeding_milestone` tasks disappear from today's breeding flow because density logic only loads due-today tasks.

**Warning signs:**
- `getHomeCards` no longer separately queries all pending `type='breeding_milestone'`.
- Future follicle/pregnancy/weaning milestones only appear in WeekStrip future dates, not today's workflow section.
- Breeding count drops after recording heat because the generated follicle task is future-dated.

**Prevention strategy:**
- Phase 1: preserve the existing test "发情后生成的未来繁育节点也应出现在今日首页流程区".
- Phase 2: define breeding workflow grouping as "active flow state", not "today due tasks".
- Phase 3: visually mark future workflow rows as upcoming/imminent rather than hiding them.

**Phase should address:** Phase 1 and Phase 2.

### Pitfall 11: Timezone Boundary Drift

**What goes wrong:** Beijing day boundaries drift around midnight, causing overdue days, medication day number, WeekStrip dots, and selected date views to disagree.

**Warning signs:**
- New code uses `new Date('YYYY-MM-DD')` without `T00:00:00+08:00`.
- Frontend and cloud each add a different `startOfDay` helper.
- Medication `currentDay` differs between homepage and health-service completion.
- Yesterday shows as overdue 2 days.

**Prevention strategy:**
- Phase 1: add UTC+8 boundary tests for overdue, medication day, and WeekStrip date counts.
- Phase 2 or 3: centralize date helper usage where practical; do not introduce additional ad hoc day math.
- Phase 4: manual UAT with timestamps near 00:00 Beijing time.

**Phase should address:** Phase 1 and Phase 4.

### Pitfall 12: URL Payload Limit from Dense Batch Navigation

**What goes wrong:** Dense health groups with many dogs navigate to forms by URL-encoding large `batchDogs` and `details` payloads, eventually exceeding platform route limits.

**Warning signs:**
- New density group passes more dog metadata than `_id`, `name`, and minimal role.
- "Process all visible" sends 30-50 dogs plus nested task details in the URL.
- Navigation randomly fails only for large groups.

**Prevention strategy:**
- Phase 2: decide whether density groups keep current URL contract or introduce a short-lived batch context token in Pinia/storage.
- Phase 3: if adding "view more/process subgroup", pass IDs or a token, not full expanded objects.
- Phase 4: UAT with 35-dog batch vaccination/deworming groups.

**Phase should address:** Phase 2 and Phase 3.

## Minor Pitfalls

### Pitfall 13: Empty and Folded States Lie

**What goes wrong:** A section says "犬群一切正常" or "无事项" while folded low-priority work exists, or after optimistic removal before backend reconciliation.

**Warning signs:**
- Empty state checks only visible expanded rows.
- Section badges show expanded count, not total available count.
- Suppressed tasks are counted in text while hidden from cards.

**Prevention strategy:**
- Phase 3: distinguish `visibleCount`, `totalCount`, and `suppressedCount` in the view model.
- Phase 4: UAT after partial completion and refresh.

**Phase should address:** Phase 3.

### Pitfall 14: Cached Smart Recommendations Use Stale Shapes

**What goes wrong:** `taskStore.buildSmartRecommendations()` receives new density group shapes and routes to wrong pages or low-value defaults.

**Warning signs:**
- FAB recommendations disappear after density work.
- First recommendation routes `breeding_milestone` to cycle detail instead of the locked record page.
- Cached cards render old grouped data after deployment.

**Prevention strategy:**
- Phase 2: keep `taskStore.cards` either on old card shape or add an adapter for recommendations.
- Phase 3: smoke test FAB recommendations after loading dense homepage data.

**Phase should address:** Phase 2 and Phase 3.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|----------------|------------|
| Phase 1 Contract Freeze | Tests only cover backend merge, not frontend reconciliation | Add focused tests/helpers for suppression, metadata sync, and route params before UI edits |
| Phase 1 Contract Freeze | Missing fixture diversity | Include mixed breeding/health/medication, partial batch, future medication-only, mixed vaccine/deworm subtype |
| Phase 2 Backend Protocol | New grouping overwrites card contract | Make workbench fields additive and preserve existing `tasks`, `sectionType`, `domain`, `priority`, `id` |
| Phase 2 Backend Protocol | Breeding grouped as batch | Group by step for display only; rows remain single workflow actions |
| Phase 2 Backend Protocol | Counts become group counts | Return separate display counts and raw actionable counts; WeekStrip uses visibility boolean/count rules |
| Phase 3 Frontend UI | Local removal duplicated per component | Centralize mutations through one page/store-level reconciliation adapter |
| Phase 3 Frontend UI | Folded groups hide urgent work | Apply domain priority scoring before fold thresholds |
| Phase 3 Frontend UI | Medication done rows dominate | Sort unfinished/partial before done; fold or weaken done rows |
| Phase 4 UAT | Race conditions missed locally | Test out-of-order refresh, immediate back navigation, date switching, and backend partial failure |
| Phase 4 UAT | 30-50 dog density not actually tested | Seed large representative data and verify first viewport, fold labels, counts, and batch navigation |
